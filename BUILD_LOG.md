## 2026-04-21 вҖ” continuation scope + ambient skip + typing delay: nexus stops feeling like a vending machine

Three fixes to the reply path, shipped together because the failure mode was one experience: nexus butting into conversations it wasn't part of. Observed in the wild вҖ” a member asked nexus a homework question, nexus answered, then two other humans started talking to each other in the same channel and nexus replied to every one of their messages within the 60s continuation window. Classic vending-machine behavior, directly counter to the "addictive not useful" lens. (1) **User-scoped continuation** in `nexus_continuation.py` вҖ” state changed from `{channel_id: timestamp}` to `{channel_id: (user_id, timestamp)}`. `mark_replied(channel_id, user_id=...)` now records WHO nexus replied to; `is_in_window(channel_id, user_id)` returns True only if the stored recipient matches the passed user_id. So when nexus replies to user A, only A's next message in that channel auto-triggers вҖ” B and C talking to each other doesn't. Window also dropped 60s вҶ’ 30s because the longer window felt like nexus was still clinging to a minute-old exchange. Scopes propagated through both call sites: `nexus_bot.on_message` direct-reply path passes the message author's id; `nexus_proactive._generate_and_send` passes the chime target's id (None means ambient chime, no recipient, window won't match any user вҖ” safe no-op). (2) **Ambient-skip die** in `nexus_bot.on_message` вҖ” when continuation is the ONLY trigger (no @, no name-trigger), roll a 30% skip probability. Real humans don't reply to every follow-up after they just said something; nexus shouldn't either. Direct @ and name-trigger always respond вҖ” the skip only fires on ambient. Debug-surface gets a new reason string `"continuation window (ambient-skipped)"` so we can see the roll in `/state`. (3) **Human-ish typing delay** before every reply send. Base 2s + 0.5s per 100 chars, capped at 6s. Wrapped in `message.channel.typing()` so Discord's native "Nexus is typingвҖҰ" indicator shows during the sleep вҖ” feels like someone read and thought, not like a vending machine instantly emitting. Added `import random as _random` to nexus_bot.py for the dice roll. Three-file diff: `nexus_continuation.py` (state shape + signature update), `nexus_bot.py` (continuation call + skip die + typing delay + mark_replied scope), `nexus_proactive.py` (pass user_id through to mark_replied). AST-verified, shipped via the now-actually-working `POST /restart` вҖ” confirmed live with `window=30s, user-scoped` in the boot log.

## 2026-04-21 вҖ” dev-loop fixes: reload-hook system + /restart fully working + logs_scoped

Three fixes in one bounce on top of the pillars wave. (1) **Post-reload hook system** in `nexus_debug_http.py` вҖ” modules that monkey-patch another module now call `register_post_reload_hook(module_name, callback)` and the /reload endpoint fires those callbacks after `importlib.reload` succeeds. Response now returns `hooks_fired` and `hooks_errors` arrays for observability. First user: `nexus_lottery.install()` registers its `_install_rare_patch` for `"nexus_mind"`, so `/reload nexus_mind` no longer silently blows away the ~1% gold wrapper. Hook errors are isolated вҖ” one bad callback doesn't abort the rest. Verified live: `/reload nexus_mind` returned `hooks_fired: ["_install_rare_patch"]`, log showed the patch re-installed. (2) **/restart now actually works.** Three bugs were stacked. First, the old handler used `threading.Thread(target=time.sleep(0.6) + os._exit(0))` which raced the aiohttp response write вҖ” clients saw `null` bodies because os._exit killed the event loop before the JSON reached the socket. Fixed by scheduling exit as an `asyncio.create_task` inside the running loop so the response flushes first. Second, `_spawn_detached_restart` was opening the log file directly from the parent Python (`open(log_path, "ab")`), which hit `PermissionError [Errno 13]` on Windows because the parent's cmd.exe wrapper already had a share-exclusive handle on the same file. Fixed by using `subprocess.Popen(["powershell.exe", "-Command", "Start-Process вҖҰ -WindowStyle Hidden"])` so the grandchild cmd.exe owns its own redirect вҖ” same pattern as `nx start`. Third, even with the new spawn, the grandchild's cmd would race the parent to open the log (parent still had it for ~1s until exit fired) and silently lose. Fixed by prefixing the grandchild's command with `timeout /t 2 /nobreak > nul &&` so it waits 2s before attempting the redirect, guaranteeing the parent has died and released the log. End-to-end verified: `POST /restart` вҶ’ proper JSON body with old/child pids вҶ’ new process alive 2s later вҶ’ old process gone вҶ’ /ping returns the new pid after boot. `/kill` got the same asyncio-task exit treatment for consistency. (3) **`POST /logs_scoped`** added to `nexus_logs_catchup.py` вҖ” takes `{user_ids: [...]}`, resets `#рҹ“қв”Ӯlogs` overwrites back to Architect-only default, then adds per-user view overrides for the named members. Used to replace the earlier broad `/logs_void` flip with a scoped grant: a member (id `[redacted]`) can now read `#рҹ“қв”Ӯlogs` as a user-level exception without getting promoted to Architect (which would also give him VoidвҶ’Signal promotion power). Other Void/Signal members lose access. Channel perms applied cleanly on first try.
